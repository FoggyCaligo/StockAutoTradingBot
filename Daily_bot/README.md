# Daily Bot

Daily Bot은 키움 REST API를 이용해 KOSPI200 후보군을 스캔하고, 20호가 기반 예상가를 바탕으로 당일 단타를 수행하는 자동매매 봇입니다.

현재 운영 기준은 다음 두 가지입니다.

- 전략 잠재수익률은 과소평가하지 않는다.
- 체결, 손익, 감사용 기록은 브로커 원본 기준으로 엄격하게 검증한다.

## 핵심 운영 시간

```yaml
market:
  prewarm_start_time: "08:55"
  start_buy_time: "09:30"
  stop_buy_time: "13:00"
  force_sell_time: "15:00"
  reconcile_time: "15:15"
  end_time: "15:20"
```

- `08:55 ~ 09:30`
  유니버스 준비와 후보 스캔용 워밍업 구간입니다.
- `09:30 ~ 13:00`
  신규 매수와 체결 기반 목표매도 관리가 이뤄집니다.
- `13:00 ~ 15:00`
  신규 매수는 중단하고 기존 보유/주문만 관리합니다.
- `15:00`
  미체결 주문을 정리하고 보유분을 강제청산합니다.
- `15:15`
  키움 브로커 체결내역과 로컬 기록을 일괄 대조합니다.
- `15:20`
  대조 완료 후 세션을 종료합니다.

## 하루 루프 개요

이 봇은 하루를 하나의 세션으로 보고, 장 시작 전 예열로 당일 운용 기준을 확정한 뒤 장중에는 같은 기준을 유지하면서 스캔, 매수, 청산, 기록 검증을 반복합니다. 핵심은 "매수 후보를 자주 다시 계산하되, 슬롯 수와 종목당 예산은 세션 시작 시점 기준으로 고정"하는 점입니다.

## 하루 루프 상세

### 1. 08:55 ~ 09:30: 예열과 세션 기준 확정

이 구간에서는 실제 매수 대신, 당일 어떤 조건으로 움직일지 준비합니다.

- 유니버스는 기본적으로 `KOSPI200`에서 출발합니다.
- 시가총액 `3000억 원` 이상, 거래대금 `50억 원` 이상 조건으로 너무 작은 종목을 먼저 제외합니다.
- `trend_filter.enabled: true`이면 최근 흐름이 너무 약한 종목을 추가로 걸러냅니다.
- 종목 목록은 우선 `data/kospi200_latest.csv`를 사용하고, 필요 시 `data/kospi200.csv`를 fallback으로 봅니다.
- 동시에 계좌 전체 자본을 추정해 당일 슬롯 수, 종목당 예산, 사실상 장중 보유 가능 종목 수를 확정합니다.

현재 슬롯 계산 기준:

```yaml
risk:
  min_slot_count: 3
  slot_budget_unit_krw: 5000000
  max_slot_count: 10
  target_budget_ratio_per_stock: 0.50
  max_budget_per_cycle_krw: 0
  max_budget_per_stock_krw: 5000000
  max_position_count: 0
```

개념적으로는 다음처럼 동작합니다.

- 최소 슬롯 수는 항상 `3개`입니다.
- `slot_budget_unit_krw`가 있으면 `전체자본 // 500만 원` 기준으로 슬롯 수를 늘립니다.
- 다만 슬롯 수는 `10개`를 상한으로 두며, 그 이상 자본 구간에서는 추가 슬롯을 만들지 않습니다.
- `10슬롯`을 넘는 자본 구간에서는 종목당 `500만 원` 상한을 해제하고 `전체자본 / 10`으로 종목당 예산을 계산합니다.
- 이렇게 확정된 슬롯 수와 종목당 예산은 장중에 현금이 잠깐 변해도 세션 기준값으로 유지합니다.

### 2. 09:30 이후: 매수 창이 열리면 루프 시작

장 시작 시간이 지나면 봇은 같은 루프를 반복합니다.

먼저 현재 보유 종목과 미체결 주문을 읽고, 장중 체결 누락이 없는지 확인합니다.

- 장중 체결은 `ka10076` 기반 조회로 로컬 DB에 반영합니다.
- 이미 보유 중이거나 미체결 주문이 있는 종목은 "활성 종목"으로 간주해 신규 후보에서 제외합니다.
- 직전 스캔에서 본 후보는 watchlist로 남겨 장중 가격/기대수익률 추적에 활용합니다.

그 다음, 리스크 제어를 먼저 점검합니다.

- 보유 포지션이 있으면 손절 감시를 우선 수행합니다.
- 손절 기준은 현재 실험상 최적선으로 보는 `매수가 대비 -4%`입니다.
- 손절 주문은 체결 가능성을 우선해 최우선 매수호가 기준의 보수적 지정가 매도로 넣습니다.
- 손절이 실행되면 그 루프에서는 추가 매수보다 체결 반영과 상태 정리에 우선순위를 둡니다.
- 계좌가 일일 손실 제한에 도달하면 신규 매수를 중단하고 관찰 모드로 넘어갑니다.

### 3. 스캔: 유니버스 전체에서 당일 후보 계산

매수 가능 시간이고, 손실 제한에도 걸리지 않았고, 빈 슬롯도 남아 있으면 스캔을 수행합니다.

스캔의 핵심은 각 종목에서 20호가 스냅샷을 한 번 읽고, 그 안에서 단기 균형 가격을 추정하는 것입니다.

- 매수/매도 잔량이 상대적으로 균형을 이루는 가격대를 찾습니다.
- 균형이 깨지는 frontier 근처를 기준으로 예상가를 계산합니다.
- 예상가는 틱 사이즈에 맞춰 반올림합니다.
- 실제 목표 매도가는 `expect_price - 1 tick`으로 둡니다.

기대수익률은 아래 식으로 계산합니다.

```text
(target_sell_price - current_price) / current_price * 100
```

### 4. 필터: 계산된 후보를 실제 매수 가능 후보로 줄이기

스캔 결과 전체를 곧바로 매수하지는 않습니다. 여러 단계 필터를 통과한 종목만 남깁니다.

현재 기본 설정:

```yaml
strategy:
  top_ratio: 0.20
  max_buy_count: 0
  min_expected_return_percent: 0.25
  max_spread_percent: 0.7
  max_prev_day_change_percent: 15.0
  sell_tick_offset: 1
  scan_interval_seconds: 60
```

개념적인 필터 순서는 다음과 같습니다.

- 먼저 계산 점수가 좋은 상위 `top_ratio` 비율만 남깁니다.
- 그 다음 기대수익률이 `0.25%` 이상인지 확인합니다.
- 스프레드가 `0.7%` 이하인지 확인합니다.
- 전일 상승률이 `+15.0%` 이상인 급등 종목은 제외합니다.
- `target_sell_price > current_price`인지 확인합니다.
- 이미 보유 중이거나 미체결 주문이 걸린 종목은 제외합니다.
- 현재 주문가능현금과 세션 종목당 예산 기준으로 최소 `1주` 이상 살 수 있는 종목만 남깁니다.

즉, "기대값이 있어 보이는 종목"과 "지금 실제로 살 수 있는 종목"을 분리해서 보고, 마지막에 둘 다 만족하는 종목만 매수 대상으로 선택합니다.

### 5. 매수 수량 결정: 빈 슬롯과 현금 제약을 동시에 반영

필터를 통과한 후보가 생기면, 이번 루프에서 몇 종목을 살지 다시 계산합니다.

- 먼저 현재 빈 슬롯 수를 계산합니다.
- 다음으로 주문가능현금을 읽어 이번 루프에서 실제로 집행 가능한 현금을 계산합니다.
- 세션 시작 때 확정한 종목당 예산을 기준으로, 빈 슬롯 수와 현금 사정을 동시에 만족하는 매수 개수를 정합니다.
- 후보가 많아도 빈 슬롯보다 많이 사지 않고, 현금이 부족하면 목표 개수보다 적게 사게 됩니다.

### 6. 매수와 즉시 매도 대기 전환

선정된 종목은 순서대로 매수 주문을 넣고, 체결이 확인되면 곧바로 목표 지정가 매도 주문을 넣습니다.

핵심 동작은 아래와 같습니다.

- 매수 체결 전에는 매도 주문을 넣지 않습니다.
- 매수 체결 직후 목표 지정가 매도 주문을 넣습니다.
- 분할체결이면 실제 체결가 기준으로 수량과 평균단가를 반영합니다.
- 지정가 매도 체결을 기다리면서도, 별도 손절 감시는 계속 돌아갑니다.

즉 이 봇은 "좋아 보이는 종목을 오래 보유"하는 구조가 아니라, "진입 직후 청산 주문까지 완성"하는 초단기 실행형 구조입니다.

### 7. 13:00 ~ 15:00: 신규 매수 중단, 보유분 관리만 수행

`13:00`이 지나면 신규 매수는 멈춥니다. 다만 세션은 끝나지 않습니다.

- 기존 포지션 손절 감시
- 지정가 매도 체결 확인
- 장중 체결 누락 보정
- watchlist 추적과 기록 저장

즉 오후 구간은 "새 기회 탐색"보다 "이미 연 포지션을 안전하게 닫는 것"이 중심입니다.

### 8. 15:00: 강제 청산

`15:00`이 되면 장중 전략은 종료 단계로 들어갑니다.

- 미체결 주문을 정리합니다.
- 남아 있는 보유 종목은 강제청산합니다.
- 강제청산 직후에도 체결 조회를 다시 돌려 로컬 기록을 최대한 맞춥니다.

이 시점부터는 더 이상 신규 진입을 하지 않고, 당일 포지션을 모두 닫는 데 집중합니다.

### 9. 15:15 ~ 15:20: 마감 대조와 기록 확정

장 마감 후에는 브로커 원본과 로컬 기록을 다시 맞추는 마감 대조를 수행합니다.

- 브로커 체결 전체를 다시 조회합니다.
- 빠진 `SELL` 체결이 있으면 채웁니다.
- 잘못 저장된 가격, 시간, source가 있으면 브로커 기준으로 덮어씁니다.
- 정정이 끝나면 `bot.sqlite3`의 `fills` 테이블을 기준으로 `fills_YYYYMMDD.csv`, `trade_fills_audit.csv`를 다시 생성합니다.

이 때문에 장중 로그보다 `15:15` 이후 정리된 기록이 더 공식적인 결과에 가깝습니다.

## 체결기록과 감사용 로그

### 원장 우선순위

신뢰 우선순위는 아래와 같습니다.

1. 키움 브로커 원본 응답
2. SQLite `fills` 테이블
3. `fills_YYYYMMDD.csv`
4. `trade_fills_audit.csv`

### 체결 저장 방식

- 장중에는 `ka10076` 체결조회로 주문별 체결을 직접 찾습니다.
- `ka10076`의 `ord_no`는 정확한 주문번호 필터가 아니라 과거 조회용 커서이므로, 당일 체결목록을 조회한 뒤 로컬에서 주문번호를 매칭합니다.
- 분할체결이면 가중평균 체결가를 저장합니다.
- 수수료/세금은 가능하면 브로커 원본 `tdy_trde_cmsn`, `tdy_trde_tax`를 사용합니다.

### 15:15 마감 대조

장 마감 후에는 브로커 체결 전체를 다시 조회해 로컬 체결 원장을 정정합니다.

- 빠진 `SELL` 체결을 채웁니다.
- 잘못 저장된 가격/시간/source를 덮어씁니다.
- 그 뒤 `fills_YYYYMMDD.csv`와 `trade_fills_audit.csv`를 DB 기준으로 다시 생성합니다.

즉, 장중 기록보다 `15:15` 이후 기록이 더 신뢰도가 높습니다.

### `trade_fills_audit.csv` 주의사항

이 파일은 감사용 누적 원장입니다.

- 종목별 누적 매수/매도/손익 상태를 담습니다.
- 가족이나 제3자가 엑셀로 필터링해 보기 좋게 만든 파일입니다.
- `SELL` 행을 단순 합산하면 종목별 누적값이 중복되어 총손익이 과대계산될 수 있습니다.

따라서 이 파일은 "거래 증빙" 용도로 좋고, "총손익 집계"는 별도 요약 로직이나 브로커 API와 함께 봐야 합니다.

## 장중 숫자와 마감 후 숫자

- 장중 손익: `MTS 우선`
- 마감 후 체결/감사 기록: `15:15 대조 후 로컬 기록 우선`

현재 운영 원칙:

- 전략 성과는 과소평가하지 않는다.
- 숫자는 MTS, 브로커 응답, DB, CSV를 서로 대조하면서 검증한다.
- 손절 기준은 현재까지의 실험 결과 기준 `-4%`를 기본 최적선으로 본다.

## 과거 성과 확인

과거 성과는 아래 API 조합으로 확인합니다.

- `ka10074`
  기간 실현손익, 매수/매도금액, 수수료, 세금
- `kt00002`
  기간별 예수금/추정예탁자산 추이
- `kt00005`
  현재 평가손익과 보유 포지션 상태
- `ka10076`
  당일 체결 상세

권장 해석:

- 기간 실현손익: `ka10074`
- 현재 실현+미실현 총손익: `ka10074 + kt00005`
- 시작자산 대비 수익률: `kt00002` 첫날 예수금 또는 추정예탁자산 기준

## 운영 팁

- `Daily_bot/logs/run_real.lock`에서 현재 실행 PID를 확인할 수 있습니다.
- 장중에는 로그보다 MTS 숫자를 우선 확인하는 편이 안전합니다.
- 마감 후 기록 검증은 `fills_YYYYMMDD.csv`, `trade_fills_audit.csv`, `bot.sqlite3`의 `fills` 테이블 순서로 확인하면 됩니다.
## 성과 측정 기준

과거 로그는 포맷 변경, 누락 복구, 재시작, 일부 기간 데이터 폐기 이력이 있으므로 장기 평균 수익률을 해석할 때 그대로 신뢰하지 않는다.

현재부터는 아래 기준을 공식 기준으로 사용한다.

1. 장중 체감 손익 확인은 `MTS`를 우선한다.
2. 당일 공식 체결 기록은 `15:15` 이후 `EOD reconciliation`이 끝난 뒤의 `bot.sqlite3` `fills` 테이블을 기준으로 본다.
3. `fills_YYYYMMDD.csv`와 `trade_fills_audit.csv`는 위 DB 기준 결과를 사람이 보기 쉽게 다시 내보낸 산출물로 본다.
4. 장기 성과 평가는 로그 형식이 고정된 이후의 연속 구간만 사용한다.
5. 일평균 수익률, 월수익률, 승률, 손익비, 최대낙폭은 최소 `20영업일` 연속 데이터가 쌓인 뒤 판단한다.
6. 중간에 로그 형식, 수익 계산 방식, 체결 복구 로직이 바뀐 경우 그 시점 전후 구간은 분리해서 해석한다.

즉, 앞으로의 공식 성과 평가는 아래 순서를 따른다.

1. 장중: `MTS`로 체감 손익 확인
2. 마감 후: `15:15` 이후 `bot.sqlite3` `fills` 기준 확정
3. 검산: `fills_YYYYMMDD.csv`, `trade_fills_audit.csv`와 대조
4. 통계 평가: 포맷 고정 이후 `20영업일` 이상 연속 데이터만 사용

현재 판단 원칙:

- `일 0.15%` 목표는 가능성이 있지만, 연속 실측 데이터가 더 쌓이기 전까지는 확정 평균으로 단정하지 않는다.
- 반대로, 과거 일부 로그만으로 전략 수익률을 과소평가하지도 않는다.
- 전략의 기대값과 로그의 신뢰도는 분리해서 해석한다.
