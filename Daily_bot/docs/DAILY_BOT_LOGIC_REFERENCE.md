# Daily Bot Logic Reference

이 문서는 현재 실코드와 백테스트 코드 기준의 핵심 동작을 개념 위주로 정리한 문서다. 진실 원천은 [main.py](/C:/Users/bigla/OneDrive/Documents/GIT/StockAutoTradingBot/Daily_bot/main.py), [settings.yaml](/C:/Users/bigla/OneDrive/Documents/GIT/StockAutoTradingBot/Daily_bot/config/settings.yaml), [replay_market_traces.py](/C:/Users/bigla/OneDrive/Documents/GIT/StockAutoTradingBot/Daily_bot/backtest/replay_market_traces.py) 이다.

## 1. 전략 목표

데일리 봇은 장중 짧은 구간의 미세한 기대수익 기회를 찾고, 조건을 만족한 종목만 빠르게 진입한 뒤 빠르게 정리하는 intraday 전략이다.

핵심 철학은 다음과 같다.

- 유동성이 충분한 대형주 유니버스 안에서만 탐색한다.
- 20호가 기반 예측가로 기대수익률을 계산한다.
- 기대수익률 문턱을 넘는 후보만 진입 대상으로 본다.
- 진입 직후 목표가 매도 주문을 걸어 짧은 익절을 우선한다.
- 손절, 일손실 제한, 강제청산으로 계좌 리스크를 제한한다.

## 2. 유니버스 구성

현재 실코드는 고정 CSV 유니버스가 아니라 당일 조회되는 전체 KOSPI 종목을 기준으로 유니버스를 만든다.

기본 필터는 다음과 같다.

- 시장: `KOSPI`
- 시가총액 하한
- 거래대금 하한
- 추세 필터는 코드에 있으나 현재 운영값은 `OFF`

유니버스 결과는 캐시 파일로 남길 수 있지만, 기준 자체는 당일 전체 KOSPI 조회다.

## 3. 스캔 주기와 배치 개념

실거래는 `scan_interval_seconds = 60` 기준으로 스캔한다.

중요한 점은 “개별 행 단위”가 아니라 “한 번의 스캔 배치”가 의사결정 단위라는 것이다.

- 같은 스캔 배치 안에서 유니버스 전체를 순회한다.
- 각 종목의 20호가 스냅샷을 저장한다.
- 모든 후보를 계산한 뒤, 그 배치 전체를 한 번에 필터링한다.
- 진입 여부는 배치 전체 후보군을 본 뒤 결정한다.

백테스트도 현재는 `scan_cycle_at` 기준으로 같은 배치를 묶어서 판단하도록 맞춰져 있다.

## 4. 예측가와 기대수익 계산

각 종목에 대해 다음 흐름으로 기대수익을 만든다.

1. 20호가 조회
2. 호가 구조로부터 예측가(`expect_price`) 계산
3. `sell_tick_offset` 을 반영한 목표 매도가 계산
4. 현재가 대비 기대수익률(`expect_revenue_percent`) 계산
5. 스프레드와 상위 5호가 매도잔량 금액도 함께 계산

개념식은 다음과 같다.

```text
expect_revenue_percent = (target_sell_price - current_price) / current_price * 100
```

또한 목표 매도가가 진입가보다 낮아지는 비정상 상황을 막기 위해, 최소 1틱 위 가격을 강제로 보장한다.

## 5. 후보 필터

현재 실코드의 후보 필터는 아래 요소들로 구성된다.

- `top_ratio`
- 최소 기대수익률 문턱
- fallback 기대수익률 문턱
- 스프레드 상한
- 전일 종가 대비 상승/하락률 범위
- 직전 스캔 대비 급등 필터
- 이미 보유 중인 종목 제외
- 추세 필터 통과 여부

현재 운영 설정에서 실제로 켜져 있는 것은 다음이다.

- `top_ratio = 1.0`
- `min_expected_return_percent = 0.7`
- fallback: `[0.6, 0.5]`

현재 운영 설정에서 꺼져 있는 것은 다음이다.

- 스프레드 상한 필터
- 전일 변동률 범위 필터
- 직전 스캔 급등 필터
- 매도호가 잔량 비율 필터
- 추세 필터

즉 현재는 기대수익률 문턱이 사실상 가장 강한 진입 필터다.

## 6. fallback 동작

fallback 은 항상 쓰는 것이 아니다.

- 먼저 기본 기대수익률 문턱(`0.7`)으로 후보를 고른다.
- 후보가 0개이고, 현재 활성 포지션도 없는 “완전 빈 배치”일 때만 fallback 을 시도한다.
- 순서는 `0.6 -> 0.5`
- 중간에 후보가 생기면 그 문턱에서 멈춘다.

즉 보유 중이거나 이미 후보가 있는 배치에는 fallback 이 개입하지 않는다.

## 7. 자본 계획과 슬롯

이 전략은 고정 3종목 전략이 아니다.

실제 동작은 자본 기반 슬롯 계산을 따른다.

- 최소 슬롯 수
- 최대 슬롯 수
- 슬롯 예산 단위(`slot_budget_unit_krw`)
- 종목당 목표 예산 비율
- 종목 수 하드 상한(`max_position_count`)

현재 운영값 기준으로는 자본이 커질수록 슬롯 수가 늘어날 수 있고, 총 보유 종목 수 상한은 최대 10개까지 열려 있다.

다만 `strategy.max_buy_count = 3` 은 여전히 중요하다.

- 이것은 “한 번의 신규 진입 배치에서 동시에 추가로 살 수 있는 최대 종목 수”다.
- 총 보유 한도와는 다른 개념이다.

## 8. 빈 슬롯 재진입 정책

현재 기본 실거래 동작은 보수적이다.

- 배치로 몇 종목을 산 뒤
- 그중 일부가 먼저 팔려 빈 슬롯이 생겨도
- 남은 종목과 미체결 주문이 모두 정리되어 배치 전체가 완전히 비기 전까지
- 새 후보를 다시 사지 않는다

코드상 개념은 “full batch exit 을 기다린다”에 가깝다.

백테스트도 기본값은 같은 동작이다.

- 기본: `disallow-refill-empty-slots`
- 옵션으로만 `allow-refill-empty-slots`

## 9. 실제 매수 선택 방식

진입 후보가 생겼다고 상위 N개를 단순히 자르는 것이 아니다.

실코드는 `select_affordable_targets` 로 실제로 살 수 있는 조합을 고른다.

여기서 보는 요소는 다음과 같다.

- 목표 종목 수
- 남은 현금
- 남은 슬롯 수
- 종목별 균등 예산 분배
- 실제 수량 계산 가능 여부
- 목표가가 진입가보다 유효하게 높은지
- 매도호가 잔량 비율 제한이 켜져 있으면 그 제한 통과 여부

중요한 결과는 다음과 같다.

- 기대수익이 높은 종목이 있어도 예산 구조상 비효율적이면 제외될 수 있다.
- 비싼 후보 하나보다, 더 싼 후보 2개 조합이 선택될 수 있다.
- 따라서 “랭킹 순위 = 실제 매수 순위”는 아니다.

백테스트도 현재 이 조합 선택 로직을 동일하게 사용한다.

## 10. 매수 실행

실거래 매수 흐름은 다음과 같다.

1. 배치 후보 스캔
2. 필터 통과 후보 계산
3. 빈 슬롯 수와 주문 가능 현금 확인
4. 실제로 살 수 있는 조합 선택
5. 선택된 종목만 `signals(selected=True)` 와 `market_traces` 에 남김
6. 실제 매수 주문 전송

매수 직후에는 체결가를 기준으로 바로 익절 계획과 손절 계획이 정해진다.

## 11. 익절 로직

익절은 기본적으로 짧은 목표가 청산이다.

- 체결 직후 목표 매도가를 정한다.
- 목표가는 `expect_price` 와 `sell_tick_offset` 에서 계산된다.
- 최소 1틱 이익은 강제로 보장한다.
- 실제 주문은 진입 직후 바로 매도 대기 상태로 들어간다.

실거래에서 실현수익률이 딱 `0.7%` 로만 모이지 않는 이유는 다음 요소들이 섞이기 때문이다.

- 틱 단위 반올림
- 실제 체결가와 계획 진입가 차이
- 목표가 도달 시점의 체결 구조
- 부분체결/분할청산

## 12. 손절 로직

현재 운영 손절은 퍼센트 손절이 기본이다.

- `stop_loss_percent = 4.5`
- `stop_loss_tick_count = 0`
- `stop_loss_tick_multiplier = 0.0`

즉 현재 운영값에서는 틱 기반 동적 손절이 꺼져 있고, `-4.5%` 손절이 중심이다.

손절이 발생하면 기존 매도 주문을 정리하고 손절 매도를 우선한다.

## 13. 일손실 제한

봇은 개별 종목 손절만 보는 것이 아니라, 계좌 전체의 일손실 제한도 본다.

- 시작 계좌가치 대비 손실률을 계산한다.
- `daily_loss_limit_percent` 를 넘으면 추가 진입을 막고 방어 모드로 들어간다.

즉 전략은 종목 단위와 계좌 단위의 리스크 방어를 둘 다 가진다.

## 14. 강제청산과 장 마감 정리

장중에 정리되지 않은 포지션은 `force_sell_time = 15:00` 이후 강제 정리 대상이 된다.

이후 `reconcile_time = 15:15` 에는 브로커 체결 내역을 다시 확인하고, 누락된 fill 정보를 보정한다.

즉 장 종료 전후에는 “전략적 청산”보다 “실제 브로커 체결 정합성 복구”가 더 중요해진다.

## 15. 기록 구조

주요 기록 테이블/CSV 는 다음과 같다.

- `market_traces`
- `signals`
- `orders`
- `fills`
- `account_traces`
- `daily_reference_prices`

특히 `market_traces` 는 단순 후보 로그가 아니라, 다음 분석의 핵심 재료다.

- 어떤 배치에서 어떤 후보가 있었는지
- 어떤 종목이 실제 선택됐는지
- 보유 모니터링 상태가 어땠는지
- 백테스트 리플레이에서 같은 배치를 어떻게 재구성할지

## 16. 백테스트의 현재 재현 수준

현재 리플레이는 예전보다 실거래에 더 가깝도록 맞춰져 있다.

- `scan_cycle_at` 기준 배치 재구성
- 신규 진입은 `scan_candidate` 배치만 사용
- fallback 규칙 일치
- 직전 스캔 급등 필터 반영
- `select_affordable_targets` 사용
- 목표가 초과 시 익절 체결가는 `target_price` 로 고정

하지만 아직 완전히 같지는 않다.

남은 차이의 대표 원인은 다음과 같다.

- 60초 스냅샷 사이에서의 순간 고가/저가
- 부분체결과 분할청산
- 주문 취소 후 재주문
- 브로커 내부 체결 순서
- 기록에는 남지만 백테스트에 직접 쓰지 않는 실거래 상태 정보

즉 현재 백테스트는 “설정 변경 시나리오 실험”과 “실거래와 유사한 구조 재현” 사이의 균형을 맞춘 상태다.

## 17. 실거래와 백테스트를 볼 때의 해석 순서

둘이 다르면 아래 순서로 보는 것이 가장 정확하다.

1. 같은 날짜 로그가 모두 있는지
2. 같은 시작자본인지
3. fallback 사용 여부가 같은지
4. refill 허용 여부가 같은지
5. 첫 진입 배치에서 후보 집합이 같은지
6. 실제 선택 조합과 백테스트 선택 조합이 같은지
7. 익절/손절/강제청산 시점이 같은지
8. 부분체결과 장중 순간 고가/저가 차이가 개입했는지

## 18. 현재 한 줄 요약

현재 데일리 봇은 “당일 KOSPI 전체 대형주 유니버스를 60초마다 스캔하고, 20호가 기반 기대수익률이 충분한 후보만 고른 뒤, 자본 기반 슬롯과 실제 매수 가능 조합을 계산해서 짧게 진입하고 짧게 청산하는 intraday 배치형 전략”이다.
